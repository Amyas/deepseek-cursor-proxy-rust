# DeepSeek Cursor Proxy Rust 重构施工文档

## 1. 文档目标

本文档用于指导 `deepseek-cursor-proxy` 的 Rust 全量重构工作，目标是：

- 在保持现有 Python 版本协议行为兼容的前提下，完成 Rust 版本替换。
- 将当前项目中耦合较重的请求改写、流式处理、缓存、追踪等能力拆分为清晰模块。
- 建立明确的阶段目标、实现方案、验收标准，便于分阶段施工和阶段性验收。

本文档默认重构后的项目目录为：

- `/Users/amyas/github/deepseek-cursor-proxy-rust`

---

## 2. 重构总目标

Rust 版本需要完整覆盖当前 Python 项目的核心能力：

- OpenAI 兼容的 `/v1/chat/completions` 代理能力
- `reasoning_content` 自动恢复与补写
- DeepSeek 请求字段规范化和兼容性修复
- 非流式响应重写
- 流式 SSE 响应重写
- reasoning cache 持久化
- trace 调试落盘
- CLI + 配置文件
- 纯本地运行，不依赖 ngrok 或其他公网隧道

同时实现以下工程目标：

- 模块边界清晰，业务逻辑可单测
- 高并发下 streaming 更稳定
- 错误处理统一
- 便于后续扩展到其他本地客户端和其他上游模型供应商

## 3. 总体设计方案

建议技术栈：

- Runtime: `tokio`
- Web: `axum`
- Upstream HTTP: `reqwest`
- Serialization: `serde` `serde_json` `serde_yaml`
- CLI: `clap`
- Storage: `rusqlite`
- Logging: `tracing` `tracing-subscriber`
- Error: `thiserror` `anyhow`
- Hash: `sha2`
- Compression: `flate2`

建议目录结构：

```text
deepseek-cursor-proxy-rust/
  Cargo.toml
  src/
    main.rs
    lib.rs
    cli.rs
    config.rs
    error.rs
    app/
      mod.rs
      state.rs
    http/
      mod.rs
      routes.rs
      handlers.rs
      upstream.rs
      sse.rs
    protocol/
      mod.rs
      model.rs
      normalize.rs
      transform.rs
      response_rewrite.rs
      folding.rs
    reasoning/
      mod.rs
      keys.rs
      recovery.rs
      store.rs
      sqlite_store.rs
    trace/
      mod.rs
      writer.rs
      model.rs
    tests/
```

设计原则：

- HTTP 层只负责收发请求和响应
- 协议转换逻辑全部下沉到 `protocol/`
- reasoning 缓存读写统一通过 trait 抽象
- trace 为独立基础设施模块
- 所有关键业务逻辑都必须可脱离网络做单元测试

## 阶段 2：reasoning key 与缓存存储层

### 目标

- 完成 reasoning cache 的 key 体系和 SQLite 存储
- 支撑后续请求修复和流式落库

### 实现方案

1. 实现 key 生成纯函数：
   - `conversation_scope`
   - `message_signature`
   - `tool_call_signature`
   - `tool_call_ids`
   - `tool_call_names`
   - `turn_context_signature`
   - `scoped_reasoning_keys`
   - `portable_reasoning_keys`
2. 设计存储 trait：

```rust
pub trait ReasoningStore {
    fn get(&self, key: &str) -> Result<Option<String>, StorageError>;
    fn put(&self, key: &str, reasoning: &str, message_json: &str) -> Result<(), StorageError>;
    fn prune(&self) -> Result<(), StorageError>;
}
```

3. 使用 SQLite 实现：
   - 建表
   - 插入/更新
   - 过期清理
   - 最大行数清理
4. 实现 `store_assistant_message()` 和 portable alias 回填逻辑。
5. 使用内存 SQLite 或临时文件做测试。

### 验收标准

- 可以成功创建 SQLite 缓存文件
- key 生成结果稳定可复现
- 相同上下文生成相同 scope，不同上下文生成不同 scope
- 能通过多种 key 成功命中 reasoning
- prune 逻辑生效
- 单元测试覆盖关键 key 生成和存取行为

## 阶段 3：reasoning 恢复与请求修复主流程

### 目标

- 将 reasoning 恢复能力整合到请求转换主链路
- 实现 `recover` / `reject` 两种缺失策略

### 实现方案

1. 实现 assistant 消息判断逻辑：
   - 哪些消息必须带 `reasoning_content`
   - 哪些消息不需要修复
2. 为每条 assistant 历史消息生成 lookup keys。
3. 从 `ReasoningStore` 中按优先级查回 reasoning。
4. 命中后将 reasoning 补写回消息。
5. 未命中时按策略处理：
   - `reject`：返回结构化错误
   - `recover`：裁剪无法恢复的历史，添加恢复 notice
6. 记录 diagnostics，供日志和 trace 使用。

### 验收标准

- 对于缺失 `reasoning_content` 的 tool-call 历史，能从缓存成功恢复
- 在 `reject` 模式下，返回与 Python 行为一致的 409 错误
- 在 `recover` 模式下，能保留可恢复上下文并加恢复边界
- diagnostics 中包含命中类型、缺失条目、lookup keys 等信息
- 测试覆盖：
   - 正常命中
   - 全部未命中
   - 部分命中
   - recover 策略
   - reject 策略

## 阶段 5：流式 SSE 处理

### 目标

- 完成流式响应透传、增量解析、增量重写和增量落库
- 对齐 Python 版本的 SSE 行为

### 实现方案

1. 实现 SSE frame 解码器：
   - 逐行读取
   - 聚合 `data:` 事件
   - 识别 `[DONE]`
2. 实现 `StreamAccumulator`：
   - 拼接 `content`
   - 拼接 `reasoning_content`
   - 拼接 `tool_calls`
   - 记录 finish_reason
3. 实现 streaming 模式下的 reasoning 存储策略：
   - tool_call 结构已经识别时提前落库
   - finish 后再做完整落库
4. 实现下游 SSE chunk 重写和重新编码。
5. 正确处理 client disconnect、上游断流、半截 chunk 等异常。

### 验收标准

- 流式响应在 Cursor 或 curl 下可正常消费
- reasoning 内容能实时累积
- tool_calls 参数能够正确拼接
- 中途停止流式输出时，已识别 reasoning 仍可部分保留入库
- 对 `[DONE]`、异常中断、空 chunk 的行为处理正确

## 阶段 6：Cursor UI 兼容显示

### 目标

- 将 reasoning_content 显示为 Cursor 可见的 thinking 区块

### 实现方案

1. 实现 folding 模块：
   - `<details><summary>Thinking</summary>...</details>`
   - 可选 `<think>...</think>`
2. 对非流式响应，在 assistant content 中注入展示块。
3. 对流式响应，在 chunk 中动态打开和关闭 thinking block。
4. 对请求历史消息，在发往上游前移除这些 UI 镜像块。
5. 通过配置项控制是否显示 reasoning、是否折叠显示。

### 验收标准

- Cursor 侧能看到 reasoning 展示块
- 展示块不会回流污染上游消息历史
- 非流式与流式模式行为一致
- 配置关闭后不再注入展示块
- 单元测试覆盖 thinking block 的插入和剥离

## 6. 风险与控制措施

## 风险 1：协议行为不一致

风险说明：
Rust 版本在字段过滤、reasoning 恢复边界、tool_call 拼接上容易与 Python 版本产生细微偏差。

控制措施：

- 保留 Python 版本作为对照基线
- 为关键输入输出建立快照测试
- 每个阶段都做行为对齐验证

## 风险 2：流式处理复杂度高

风险说明：
SSE chunk 的半包、异常断流、tool_calls 增量拼接都容易出错。

控制措施：

- 单独做 streaming 模块，不和 HTTP handler 混写
- 为半包、空包、异常断连写专门测试
- 优先保证正确性，再做性能优化

## 风险 3：SQLite 阻塞影响吞吐

风险说明：
reasoning cache 频繁写入可能拖慢请求链路。

控制措施：

- 先用简单可靠方案落地
- 必要时用 `spawn_blocking`
- trace 写入与缓存写入尽量减小同步阻塞

## 风险 4：本地使用边界与客户端兼容性不一致

风险说明：
不同客户端对 `localhost`、`127.0.0.1`、自定义 Base URL 的支持程度不同，可能导致“代理本身可用，但目标客户端接不进去”。

控制措施：

- 先以 `curl` 和本地脚本为标准验收路径
- 尽早验证目标客户端是否支持纯本地 Base URL
- 文档中明确标注支持范围和接入前提
